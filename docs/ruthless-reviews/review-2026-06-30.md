# Ruthless Code Review — 2026-06-30

**Scope:** Full pass over the invariant-bearing paths (reconcile, registry/store,
discovery engine + scheduler, dspy/reasoner, proposal generator + engine, git +
notification providers, all `tools/*`, logging, config). Every finding below was
read at the source; the `secrets_*` path-escape was confirmed with a `pathlib`
repro.

**Stack:** Python 3.12 / FastMCP / SQLite (SQLModel) / DSPy / APScheduler / httpx /
structlog / uv.

---

## What actually holds up (verified, not assumed)

- **Curated-field protection holds.** Discovery's update branch applies only
  `provenance_updates()` (`registry/reconcile.py`), which never returns
  `display_name/category/tags/notes`. The only path that writes curated fields is
  `update_service()`, reachable solely from manual tools.
- **DSPy gates are correct direction, not inverted.** `Reasoner` discards on
  `conf < threshold`; `PatchGenerator.generate()` returns `ok=False` → a `rejected`
  Proposal *before* any Git call; `ProposalEngine.create_for_service` commits only
  when `result.ok and not proposal_dry_run`. No rejected/low-confidence/invalid-YAML
  patch can reach a Git write.
- **Deterministic detection holds.** `import dspy` appears only in
  `dspy/reasoner.py` (lazy, inside `_ensure`) and `dspy/signatures.py`. `reconcile.py`
  and the discovery sources never import it.
- **Upstream is read-only.** Traefik/Authentik clients expose only `_get`; Docker
  only `containers.list`. Non-GET verbs live only in the Gitea/GitHub *write*
  providers (intended), which never merge — only `close_pr`.
- **Every httpx client sets an explicit timeout.** No hung-upstream-forever bug.
- **Scheduler jobs are wrapped.** `run_source` catches `except Exception`, records a
  failed `DiscoveryEvent`, and never lets a pass kill the scheduler.

## 🔴 CRITICAL / BROKEN

### 1. `secrets_*` tools allow arbitrary file read **and** write via absolute paths
The only guard is `if ".." in Path(path).parts`. Absolute paths contain no `..`, and
`pathlib`'s `repo / "/etc/x"` **discards** the repo base. Confirmed:
`Path("/data/repo") / "/root/.ssh/authorized_keys"` → `/root/.ssh/authorized_keys`.

- `secrets_add("k","v","/root/.ssh/authorized_keys")` → `target.parent.mkdir(...)` +
  `target.write_text(...)` = **arbitrary file write as the server user → host takeover.**
  (`tools/secrets.py:284-306`)
- `secrets_decrypt("/etc/passwd")` / `secrets_list_keys(...)` → **arbitrary file read
  returned to the caller.** (`tools/secrets.py:253-266`, `:380-394`)

On a LAN-only, no-auth server, any client or prompt-injected agent gets host write.
**Fix:** reject absolute paths and assert
`(repo / path).resolve().is_relative_to(repo.resolve())` in one shared helper.

### 2. `main()` builds a second, parallel object graph
`_streamable_with_scheduler` (`server.py:135-148`) constructs a fresh `RegistryStore`,
`Reasoner`, `ProposalEngine`, and `DiscoveryEngine` and runs the scheduler against
*those*, while the MCP tools registered in `build_server()` use the *first* graph.
`RegistryStore` builds its own engine from a path (unlike `ProposalStore`/`HardwareStore`,
which share an injected engine), so:

- Under `REGISTRY_DB_PATH=:memory:` the two stores are **different databases** —
  discovery writes one, tools read the other. Silent divergence.
- Even on the file DB: `discovery-run-now`/`discovery-status` hit graph #1; the
  scheduler and proposal auto-create run graph #2. Debugging "status shows nothing"
  will burn hours. Plus 2× reasoners / git providers / notifiers.

**Fix:** one composition root in `build_server`; the scheduler wrapper reuses the
already-built engine.

### 3. Discovery scheduler only starts on the `streamable-http` transport
The lifespan was gutted to `yield {}` and startup moved into the monkey-patched
`run_streamable_http_async`. On `stdio` or `sse` (and CLAUDE.md advertises "stdio by
default") **no discovery ever runs** — silently. **Fix:** start the scheduler in a
transport-independent place, or fail loudly when discovery is expected but the
transport won't start it.

## 🟡 CODE SMELLS & NON-STANDARD

- **Reasoner runs synchronously inside the open SQLite write transaction.**
  `reconcile()` calls the injected `identity_resolver`/`metadata_enricher` mid-loop
  while holding a write `Session` (`registry/store.py:317-395`). Those make blocking
  litellm HTTP calls → the event loop stalls *and* SQLite's write lock is held across
  every LLM round-trip. Off by default, but a latency landmine the moment
  `DSPY_ENABLED=true`. Resolve identity/metadata in a pre-pass.
- **Reconciler race is prevented only by coincidence** — safe purely because
  `reconcile()` is sync and never `await`s mid-body on a single-threaded loop. Make it
  async/threaded and you get duplicate-name `IntegrityError`s. (Currently contained:
  `run_source` would catch and self-heal next pass.)
- **`_scrub_credentials` only scrubs `patch`, not the model-generated
  `commit_message`/`pr_title`/`pr_body`** — all pushed to Git (`proposal/generator.py:127`
  vs `:159-161`). A secret echoed into the PR body reaches the remote unscrubbed.
- **Authentik pagination is dropped.** `_results()` returns page 1 only
  (`integrations/authentik/client.py:62-66`). `list_events` silently truncates to the
  most recent ~100, so `authentik_summarize_events` reasons over partial data and
  reports wrong counts.
- **`secrets_add` docstring lies.** It claims "staged but NOT committed," but for a new
  path it calls `secrets_encrypt`, which `git commit`s `.gitattributes`
  (`tools/secrets.py:231`, `:292`).
- **`service_link_authentik` writes a *provenance* field with no guard.** It sets
  `authentik_app_slug` and claims "Overrides discovery," but the next pass's
  `provenance_updates` clobbers it — unlike `hardware-link-service`, which sets
  `manual_link=True`.
- **Engine returns shape-varying `dict[str, Any]`**
  (`{"error"}|{"skipped"}|{"rejected"}|{"dry_run"}|proposal-dump`,
  `proposal/engine.py:91-205`). Union-by-dict-shape leaking from the business layer.
- **Hand-rolled `_migrate` with magic SQL strings** (`registry/store.py:79-101`); loop
  var `col` shadows the imported `col`. No migration tool.
- **Naive/aware datetime mismatch.** Models use `datetime.now(UTC)` (aware), SQLite
  returns naive; `_age_days` (`proposal/engine.py:235-240`) works around it case-by-case.
  `_branch_name` uses naive `datetime.now()` while everything else is UTC.
- **Per-request httpx client creation** in every client/provider (new `AsyncClient` per
  call, even per retry) — no connection reuse.
- **Broad excepts lose tracebacks.** `run_source`, the reasoner ops, and proposal hooks
  log only `str(exc)`. A bad `DSPY_API_KEY` makes every reasoning call warn-and-return-
  None forever; discovery silently stays deterministic with no loud signal.
- `_OPEN_STATUSES` (`proposal/store.py:11`) is dead. `id` shadows the builtin across
  most tools. Two separate `utcnow`/`_utcnow` helpers in the two model modules.

## 🔵 MCP PROTOCOL & CONVENTION VIOLATIONS

- **Tool naming contradicts the documented rule.** Only `hardware-*` uses
  `@mcp.tool(name="hardware-...")`. `registry_*`, `secrets_*`, `proposal_*`, `service_*`
  all use bare `@mcp.tool()` → the snake_case function name becomes the tool name.
  CLAUDE.md's "kebab-case for tool names" is honored by exactly one module. Either fix
  the tools or fix the doc.
- **Redaction omits the bare `key` substring** that CLAUDE.md/spec require.
  `_REDACT_SUBSTRINGS` has `api_key`/`apikey` but not `key` (`logging/events.py:19`), so a
  field literally named `key=` or `git_crypt_key=` is **not** redacted
  (`secrets_key_path` only survives because it contains "secret").
- **Secrets in `Settings` are plain `str`, not `SecretStr`** (`config.py`: `git_token`,
  `authentik_token`, `dspy_api_key`, `notification_token`, `secrets_git_crypt_key`). Any
  `settings.model_dump()`/`repr` into a log leaks them; the field-name redactor won't
  catch them once nested in a dict value.
- **`error=str(exc)` is logged under a non-secret key**, so the redactor can't catch a
  credential embedded in an upstream error string.
- **Direct `os.getenv` bypassing `Settings` (×3):** `TRAEFIK_MIDDLEWARE_PATH`
  (`proposal/generator.py:29`) and `DISCOVERY_EXCLUDE_NAMES` read **at import time** in
  both `discovery/traefik.py:48` and `discovery/authentik.py:22` — the latter isn't even
  a `Settings` field, so it's invisible to the config surface and frozen at import.

## 🟢 ARCHITECTURAL REFACTOR

- **Collapse the dual object graph** (the #2 critical) into one composition root in
  `build_server`; the scheduler wrapper takes the already-built engine. Highest-leverage
  cleanup for solo maintainability.
- **Give secrets a real service layer.** Every other domain has a `*Store`; secrets jams
  all orchestration (guard→repo→unlock→fs→git) inside `@mcp.tool` closures over a
  module-level `_run`, so it's untestable without standing up FastMCP and unmockable
  subprocess. Extract `SecretsService(settings, runner=_run)`; tools become 3-line
  adapters; path-confinement lives in one guarded helper.
- **Move reasoning out of the DB transaction.** Resolve identity/metadata in a pre-pass,
  hand `reconcile` plain dicts; keeps the deterministic core's session short and the loop
  unblocked.
- **Add a path-confinement helper** used by every secrets tool (and anywhere taking a
  caller-supplied path).
- **Delete the FastMCP monkey-patch** the moment you bump the SDK — it's the worst
  "what is this" trap in the repo for a future-you.

## Solo Dev Verdict

The *core* is genuinely good — the deterministic/reasoning separation is disciplined,
curated-field protection and the DSPy→PR gates actually hold, providers are cleanly
injectable, and the discovery loop is crash-safe. But there are real landmines. The
`secrets_*` absolute-path hole is a host-takeover bug sitting behind an unauthenticated
LAN tool — that's not "tech debt," that's a six-months-from-now incident. The duplicated
object graph in `main()` is the kind of thing that, after a gap, makes you stare at "why
is discovery-status empty" until you rediscover there are two of everything. Neither is
hard to fix, and nothing requires a rewrite — but **do not consider this safe to walk
away from until at least the path confinement and the dual-graph collapse are done.** Fix
those two and it's a maintainable one-person system; ship it as-is and the homelab is one
malformed tool call away from a bad day.

---

## Recommended remediation order

1. **Path confinement in `secrets_*`** (CRITICAL host write/read). Smallest, highest risk.
2. **Collapse the dual object graph** in `server.py`; verify the scheduler starts on the
   intended transport.
3. **Redaction `key` substring** + move secrets to `SecretStr`.
4. **Scrub `commit_message`/`pr_body`/`pr_title`**, not just `patch`.
5. **Tool-naming consistency** (rename to kebab or correct the convention doc).
6. **`os.getenv` → `Settings`**; Authentik pagination; reasoner-outside-transaction.
